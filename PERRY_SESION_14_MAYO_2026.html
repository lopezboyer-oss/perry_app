Decisión Arquitectónica Clave

🔄 Cancelación: Snapshot & Restore vs. Soft-Delete

Se optó por un modelo de snapshot + hard-delete para las asignaciones de recursos: al cancelar, se guarda un JSON con todas las asignaciones originales en cancelledResources, luego se borran las filas de asignación para liberar los recursos inmediatamente. Al deshacer, se parsea el JSON y se intentan re-crear las asignaciones, reportando conflictos si el recurso ya fue asignado a otra actividad. Este enfoque es preferible a soft-delete porque: (1) las queries de disponibilidad siguen siendo simples (WHERE activityId = ?), (2) no hay riesgo de "recursos fantasma" y (3) el snapshot JSON es auto-contenido para auditoría.
